# 摸鱼统计器 PRD：每日群聊战报

## 1. 文档信息

- 产品：`摸鱼统计器`
- 模块：`每日群聊战报`
- 版本：`PRD v1.0`
- 日期：`2026-03-09`
- 状态：`P0 / P1 已完成，可继续进入 P2`

## 2. 产品背景

`摸鱼统计器` 当前已经完成基础闭环：统计输入事件、日结计算、生成海报、查看历史记录。

但从产品体验上看，当前版本更像“会生成分数的桌面工具”，缺少一个能驱动用户持续打开和主动传播的核心时刻。尤其是现有海报内容过于单薄，无法支撑用户在群聊中主动转发。

本次 PRD 的目标是围绕 `分享感` 重构结果体验，让“今日战报”成为产品的核心输出。

## 2.1 当前实现快照

截至 `2026-03-09`，以下能力已经落地：

1. 每日记录模型已包含战报所需统计字段。
2. 已实现随机毒舌文案和亮点摘要生成。
3. 已实现战报图 1.0 及主窗口、菜单栏中的今日战报入口。
4. 历史页已支持回看过往称号、判词，并重新复制某天战报。
5. 已支持刷新判词。
6. 日结通知已改为结果化文案，并可通过通知点击唤起今日战报。
7. 复制今日战报时，已支持复制 `图片 + 分享文案`。

仍待继续的方向：

1. 更强的社群分享模板和文案样式。
2. 周报 / 月报纪念海报。
3. 更完整的通知权限引导和首次分享提示。

## 3. 问题定义

### 3.1 当前痛点

1. 海报只展示标题、分数和固定文案，信息密度太低。
2. 分享动作埋得太深，菜单项缺少结果感。
3. 历史页无法回看“当日状态”，只能看分数列表。
4. 主窗口没有承担结果展示职责，用户缺少“今天值不值得晒”的感知。
5. 文案固定，几天后缺乏新鲜感。

### 3.2 核心机会

如果产品能在日结后自动生成一张 `有梗、有对比、有证据` 的群聊战报卡，用户更可能：

1. 主动打开应用查看结果
2. 把战报发到相关社群
3. 把每天结果当作纪念卡片保存

## 4. 产品目标

### 4.1 业务目标

1. 让“今日战报”成为产品的核心结果输出。
2. 提升用户日结后的查看和转发意愿。
3. 在不引入联网和 AI 的前提下，做出足够有趣的分享内容。

### 4.2 用户目标

1. 用户能快速知道“我今天是什么状态”。
2. 用户能一键复制/保存一张适合发群的战报图。
3. 用户回头看历史时，能感受到战绩和纪念价值。

## 5. 目标用户与使用场景

### 5.1 目标用户

1. 办公室白领
2. 学生
3. 喜欢玩梗、自我记录、轻量晒结果的用户

### 5.2 典型场景

1. 下班前看到今日结算结果，直接复制发到工作群或朋友群。
2. 晚上回看今天状态，把战报保存到本地做纪念。
3. 翻历史记录时，重新复制某天的离谱战报发给朋友。

## 6. 设计原则

1. `先结果，后设置`
   产品主页面优先展示结果，不让功能按钮淹没核心体验。

2. `先可传播，后可收藏`
   第一版先保证发群好用，再逐步增强长期留存价值。

3. `信息要有戏剧性`
   同一张图中同时包含称号、分数、毒舌文案和亮点摘要。

4. `保持离线和可解释`
   不依赖云端服务，结果全部由本地规则生成。

## 7. 功能范围

### 7.1 第一阶段范围

1. 今日战报数据升级
2. 随机毒舌文案池
3. 战报图 1.0
4. 今日战报入口
5. 历史页回看和再次复制

### 7.2 暂不纳入

1. 多套复杂模板系统
2. 周报/月报
3. AI 文案和 AI 海报
4. 社交平台 SDK 集成
5. 多人对比、排行榜、社群 PK

## 8. 功能需求

### FR-01 今日战报结果模型

系统在每日结算后，除现有分数和标签外，还要产出以下数据：

- `劳动分`
- `摸鱼分`
- `统计分钟数`
- `活跃分钟数`
- `高活跃分钟数`
- `低活跃分钟数`
- `最长沉寂分钟数`

这些字段应保存到每日记录中，供战报图、主窗口和历史页复用。

### FR-02 随机毒舌文案

系统应基于称号档位生成“今日判词”。

要求：

1. 每个档位维护独立文案池。
2. 同一条记录可以在不改变分数的前提下重新随机文案。
3. 文案必须离线可用。

### FR-03 今日亮点摘要

系统应从当日统计数据中挑选一个最值得展示的亮点，并生成摘要文本。

第一阶段候选亮点：

1. 活跃分钟极高
2. 活跃分钟极低
3. 最长沉寂时段极长
4. 劳动分极高
5. 摸鱼分极高

### FR-04 战报图 1.0

系统应生成一张适合社群传播的竖版战报图，至少包含：

1. 称号
2. 主分数
3. 今日判词
4. 战绩摘要
5. 今日亮点
6. 日期
7. 产品名

### FR-05 今日战报入口

主窗口和菜单栏都应提供清晰的今日战报入口。

要求：

1. 如果今日已结算，优先展示战报摘要。
2. 用户可一键复制战报图。
3. 用户可保存战报图到本地。
4. 用户可刷新文案，但不能改变基础分数。

补充说明：

- 当前实现中，复制行为已经升级为复制 `战报图片 + 分享文案` 的组合。

### FR-06 历史回看

历史页应支持回看过往战报核心信息：

1. 日期
2. 称号
3. 分数
4. 判词
5. 重新复制该日战报

### FR-07 通知升级

日结完成后的通知应体现结果感，并可引导用户打开今日战报。

补充说明：

- 当前实现中，通知点击已支持唤起主窗口中的今日战报入口。

## 9. 信息结构

### 9.1 战报图信息层级

1. `身份层`：今日称号
2. `结果层`：主分数
3. `吐槽层`：随机毒舌文案
4. `战绩层`：活跃分钟、最长沉寂、统计范围、亮点摘要
5. `纪念层`：日期与产品标识

### 9.2 页面入口

1. 主窗口首页
2. 菜单栏快捷入口
3. 日结通知入口
4. 历史页回看入口

## 10. 关键交互流程

### 10.1 今日已结算

1. 系统结算完成
2. 生成今日战报展示模型
3. 通知提醒用户“今日已结算”
4. 用户打开主窗口或菜单栏
5. 用户复制或保存战报图
6. 用户可刷新一条新的毒舌文案后再分享

### 10.2 历史回看

1. 用户打开历史页
2. 选择某一天记录
3. 查看该日称号、分数和判词
4. 点击重新复制该日战报

## 11. 非功能需求

1. 全链路离线可用
2. 不新增任何隐私敏感采集
3. 同一份记录反复生成战报时，分数和统计信息保持一致
4. 战报图生成耗时应保持在用户可感知的即时范围内

## 12. 成功指标

第一阶段上线后，建议跟踪以下产品指标：

1. `战报生成成功率`
2. `复制战报图触发率`
3. `保存战报图触发率`
4. `历史回看触发率`
5. `文案重刷触发率`

当前版本暂无埋点，可先用本地调试日志和用户访谈观察。

## 13. 验收标准

1. 结算后能够稳定生成包含完整信息的战报图。
2. 用户在 2 步内可完成复制今日战报。
3. 用户可重新随机判词但不改变分数和亮点数据。
4. 历史页能够重新复制任意一条已保存记录的战报。
5. 所有功能在离线状态下可正常运行。

## 15. 后续优化建议

除本次 PRD 覆盖内容外，项目还值得继续优化：

1. 权限引导体验更友好，加入更生活化解释。
2. 设置页改用具体时间控件，而不是分钟步进器。
3. 主窗口与菜单栏分工重构，减少重复。
4. 历史页加入趋势和连续记录维度。
5. 战报视觉逐步升级为更有风格的模板系统。

## 16. 下一窗口建议起点

如果在新窗口继续，建议优先从以下两个方向选一个：

1. `P2-模板系统`
   把当前单一战报卡扩展为 2 到 3 套可切换的社群分享模板。

2. `P2-周报/月报`
   用现有 `DailyRecord` 数据生成更适合收藏的纪念海报。
