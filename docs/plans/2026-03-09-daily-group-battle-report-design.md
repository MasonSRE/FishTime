# 每日群聊战报设计稿

## 1. 目标

把 `摸鱼统计器` 从“能生成分数的统计工具”提升为“每天都想发群的结果型产品”。

本轮设计聚焦 `产品功能体验`，并且将“有分享感”定为最高优先级。目标不是单纯重画海报，而是把产品主流程升级为：

`统计 -> 结算 -> 看到战报 -> 一键复制/保存 -> 愿意发群`

## 2. 当前项目现状与问题

基于现有代码和界面结构，当前版本已经完成 MVP 闭环，但分享体验明显偏弱：

1. 主窗口和菜单栏功能重复，像操作面板，不像结果页。
2. 海报生成能力过于基础，仅有纯色背景、标题、分数和一句固定文案。
3. 历史页只有日期和分数，无法回看“当日状态”。
4. 权限引导和设置页偏技术导向，缺少情绪价值。
5. 分享行为只是菜单项之一，没有被提升为产品主线。

## 3. 已评估方案

### 方案 A：标准群聊战报图

- 目标：最快建立“看完就能发群”的分享闭环。
- 核心：称号、主分数、随机毒舌文案、亮点摘要、战绩信息。
- 优势：落地快、传播直观、与现有架构兼容度高。
- 风险：如果文案和视觉变化太少，几天后容易疲劳。

### 方案 B：模板化梗图战报

- 目标：强化传播性和识别度。
- 核心：在战报图基础上引入多套模板，例如“奖状”“通缉令”“工位日报”。
- 优势：社群传播感强，更容易形成产品特色。
- 风险：视觉和资产工作量较大，第一版容易过度设计。

### 方案 C：纪念型收藏海报

- 目标：增强“自己保存做纪念”的长期价值。
- 核心：加入周趋势、连续天数、历史最佳/最差等信息。
- 优势：留存价值高，适合个人收藏。
- 风险：群传播刺激不足，不符合当前首要目标。

## 4. 最终选择

选用 `方案 A：标准群聊战报图` 作为第一阶段主线，并吸收 `方案 B` 中的随机毒舌文案能力，但暂不引入完整多模板系统。

选择理由：

1. 最契合“群传播优先、个人保存次之”的目标。
2. 能在现有数据和本地渲染架构上快速落地。
3. 可以先验证分享链路，再决定是否继续扩展模板系统。

## 5. 核心设计

### 5.1 产品定位

`每日群聊战报图` 将成为产品的核心结果页，不再只是一个导出功能。

它同时承担两种角色：

1. `社群传播内容`：3 秒内看懂，具备对比感和戏剧性。
2. `个人纪念卡片`：能让用户回头看见“那天是什么状态”。

### 5.2 战报图信息结构

战报图固定为单张竖版结果卡，按以下顺序组织：

1. `身份层`
   - 今日称号，例如“摸鱼大师”“平衡人类”“顶级牛马”
2. `结果层`
   - 主分数，优先突出劳动分或摸鱼分
3. `吐槽层`
   - 随机毒舌文案，也可称为“今日判词”
4. `战绩层`
   - 活跃分钟
   - 最长沉寂时段
   - 统计范围
   - 今日亮点摘要
5. `纪念层`
   - 日期、产品名、适合转发或留存的底部标识

### 5.3 文案机制

第一阶段采用纯规则生成，不引入 AI，也不依赖联网。

文案分为三层：

1. `结果分类`
   - 由现有分数区间决定当天的称号和主色调。
2. `随机毒舌文案`
   - 每个称号档位准备一组文案池，战报生成时随机取一条。
3. `今日亮点`
   - 从当日统计指标中选出最值得晒的一个点，生成一句摘要。

这样既能保证产品有稳定输出，也能避免连续多天的重复感。

### 5.4 亮点摘要候选

第一阶段优先支持以下亮点：

1. 活跃分钟偏高
2. 活跃分钟偏低
3. 最长沉寂时段很长
4. 摸鱼分特别高
5. 劳动分特别高

示例：

- “连续 96 分钟低活跃，堪称隐身办公。”
- “全天高活跃占比 62%，工位火花带闪电。”
- “工作时段内仅 41 分钟有效活跃，鱼群已成规模。”

## 6. 交互流设计

### 6.1 日结流程

系统完成结算后，应直接产出“今日战报”结果，而不是只更新一条状态文本。

通知文案建议改成：

- “你今日已结算：摸鱼大师，摸鱼分 78。”

点击通知后直接进入当日战报视图。

### 6.2 主窗口

主窗口从“按钮面板”改为“今日战报首页”：

- 若今日已结算：
  - 展示称号、主分数、毒舌文案、亮点摘要
  - 提供 `复制战报图`、`保存到本地`、`重新生成文案`
- 若今日未结算：
  - 展示进行中状态、当前事件数、预估倾向

### 6.3 菜单栏

菜单栏保留最短分享链路：

1. 今日状态
2. 打开今日战报
3. 复制战报图
4. 设置
5. 历史

### 6.4 历史页

历史页从分数列表升级为“过往战报索引”：

- 展示日期、称号、分数、当日判词
- 支持重新复制某天的战报
- 后续可扩展为缩略图卡片流

### 6.5 重新生成文案

允许对同一份数据重新随机一条毒舌文案，但不改变分数和基础战绩。

这个能力实现成本低，却能显著提升分享前的可玩性。

## 7. 数据和技术影响

现有 `DailyRecord` 字段不足以支撑分享体验，需要补充可视化摘要所需数据，例如：

- `moyuScore`
- `trackedMinutes`
- `activeMinutes`
- `highActivityMinutes`
- `lowActivityMinutes`
- `longestIdleMinutes`

同时需要新增一个面向展示层的“战报组合器”，负责：

1. 根据称号选择文案池
2. 从数据中挑选亮点摘要
3. 输出可供海报和视图共用的展示模型

## 8. 第一阶段范围

### In Scope

1. 结算结果数据升级
2. 随机毒舌文案池
3. 战报图 1.0
4. 今日战报入口
5. 历史页回看和再次复制

### Out of Scope

1. 多套完整海报模板
2. 周报和月报
3. 社交平台直连分享
4. AI 文案和 AI 海报
5. 排行榜、社群 PK、多人对比

## 9. 验收标准

1. 结算后能稳定生成一张包含称号、分数、判词、亮点的战报图。
2. 同一档位的结果连续生成时，毒舌文案会随机变化。
3. 用户在 2 步以内可以完成“复制今日战报”。
4. 历史页可以重新复制过去某天的战报。
5. 整个链路保持离线可用。

## 10. 后续可优化项

本次探索中，除了“分享感”主线，还明确了这些后续产品优化方向：

1. 权限引导文案更生活化，减少“系统设置说明书”感。
2. 设置页把分钟步进器改成可读时间控件。
3. 历史页增加趋势和连续记录语义。
4. 主窗口与菜单栏进行职能分工，减少重复。
5. 结果页加入更强的视觉模板和角色标签。
